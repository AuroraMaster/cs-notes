# TLS 1.3 handshake walkthrough

**Summary** — TLS 1.3 collapses key exchange and authentication into a
single round-trip, drops every weak cipher family the IETF could agree on,
and adds 0-RTT for repeat connections.
**Why I'm reading this** — to understand the exact bytes on the wire when
something fails the handshake, and to know what to look for in a `tshark`
dump.
**Status** — draft.

## The handshake at a glance

```
   client                                       server
     │ ── ClientHello ───────────────────────────► │     1 packet, ~few hundred bytes
     │   key_share, signature_algorithms,         │
     │   supported_versions[1.3], pre_shared_key   │
     │                                            │
     │ ◄── ServerHello (encrypted from here) ─────│     1 packet
     │     EncryptedExtensions                    │
     │     Certificate (or PSK accept)            │
     │     CertificateVerify                      │
     │     Finished                               │
     │                                            │
     │ ── Finished ──────────────────────────────► │     1 packet
     │ ── application data ──────────────────────► │
```

Three packets total for the handshake. Compare TLS 1.2: at least 6, and
without TFO the TCP handshake adds another RTT on top.

## Why it's a single round-trip

In 1.3 the **client speculates a key share in ClientHello**. The server
either picks one of the offered groups (most cases) or asks for a
different one (`HelloRetryRequest`, costs an extra RTT). When the server
accepts, both sides already have enough material to derive the handshake
secret and encrypt the very next message.

In 1.2 the server first picked the cipher suite, then sent a server key
exchange, then waited for the client's premaster — three sequential
back-and-forths.

## What was removed (vs 1.2)

- RSA key exchange (no forward secrecy)
- Static DH key exchange (same problem)
- CBC mode ciphers (padding oracle attacks: BEAST, Lucky 13, POODLE)
- RC4 (biased keystream)
- MD5 and SHA-1 signatures
- Compression (CRIME)
- Renegotiation (handled differently via key update)

The remaining cipher set is small and roughly:
`TLS_AES_128_GCM_SHA256`, `TLS_AES_256_GCM_SHA384`, `TLS_CHACHA20_POLY1305_SHA256`.

## 0-RTT — the trade-off you should know

If the client has a PSK (pre-shared key) from a prior session, it can send
application data piggybacked on ClientHello. **Zero round-trips** before
the request is being processed.

The catch: 0-RTT data is **replayable**. The server has no way to prove
the message is fresh, so the spec restricts 0-RTT to idempotent requests.
Browsers + servers in practice limit it to `GET` and a small number of
header bytes. If your endpoint is `POST /charge`, 0-RTT should be off.

## What I got wrong the first time

Thinking the ServerHello was still cleartext. After ServerHello, everything
the server says (including the certificate!) is encrypted with the
handshake secret. This is why TLS 1.3 connections look opaque to passive
observers earlier in the flow than 1.2.
